# PRD: Sentinel-AI — Мультиагентна екосистема кіберрозвідки та аналізу інфраструктурних ризиків

**Автор:** Gemini (в ролі AI Architect)

**Дата:** Березень 2026

**Версія:** 3.1 (Final Core Spec)

---

## 1. Огляд продукту (Product Vision)

**Sentinel-AI** — це інтелектуальна платформа наступного покоління для Cyber Threat Intelligence (CTI). Система не просто збирає новини, а діє як автономний відділ аналітики: вона самостійно шукає загрози, перевіряє їх на реальність через автономних агентів (OpenClaw), співставляє з "цифровим двійником" вашої інфраструктури в OpenSearch і видає готові рішення для бізнесу та технічних команд.

## 2. Проблематика та бізнес-цілі

* **Інформаційний шум:** Спеціалісти витрачають 80% часу на читання нерелевантних новин. Sentinel-AI автоматизує фільтрацію.
* **Відсутність контексту:** CVE сама по собі нічого не значить, якщо невідомо, чи є в нас вразливе залізо. Sentinel-AI додає контекст інфраструктури.
* **Складність оновлень:** Аналіз Release Notes — це монотонна праця. Система автоматично виявляє потенційні конфлікти (breaking changes).

## 3. Технічний стек (The Foundation)

| Рівень | Технології |
| --- | --- |
| **Backend** | Python 3.11+, FastAPI |
| **Orchestration** | **LangGraph** (циклічні графи станів) |
| **LLM Gateway** | **LiteLLM** + **OpenRouter** (підтримка будь-яких моделей: GPT-4, Claude 3.5, Llama 3, Mistral) |
| **RAG / Database** | **OpenSearch** (Векторне сховище для інфраструктури та бази знань) |
| **Autonomous Research** | **OpenClaw** (зовнішній агент для глибокого дослідження) |
| **Observability** | **Langfuse** (трасування запитів, логування думок агентів) |
| **Security** | **Guardrails AI** (валідація виходів, маскування даних) |

## 4. Функціональні вимоги (Functional Requirements)

### 4.1. Адміністративний веб-інтерфейс

* **Управління джерелами:**
* Додавання Telegram каналів (через API ID/Hash або Bot API).
* RSS/Atom стрічки (з вибором частоти опитування).
* Reddit (конкретні сабредіти) та X.com (через API або скрейпінг).
* Прямі посилання на блоги вендорів.


* **Інвентаризація (Asset Inventory):**
* Можливість імпорту CSV/JSON списків обладнання.
* Ручне додавання активів: *Назва, Вендор, Модель, Версія ПЗ, Критичність вузла.*


* **Конфігурація моделей:**
* Вибір моделі для кожного агента індивідуально (наприклад, Triage — GPT-4o-mini, Research — Claude 3.5).
* Налаштування Fallback-стратегій (якщо Cloud модель недоступна — перемикання на локальну Llama 3).



### 4.2. Механізм аналізу та обробки

* **Trend Identification:** Агенти повинні виявляти аномальну активність. Якщо про одну CVE написали 3 різні джерела протягом години — система миттєво запускає цикл аналізу.
* **Release Notes Intelligence:** Автоматичний парсинг списків змін. Виявлення фраз на кшталт "deprecated", "security fix", "vulnerability".
* **OpenClaw Bridge:** Коли система бачить критичну загрозу, вона формує "Research Task" і відправляє його в OpenClaw для автономного пошуку деталей (PoC, методи детекції).

### 4.3. Звітність та оповіщення

* **Operations Report:** Технічний опис, посилання на GitHub з експлойтом, CLI команди для перевірки версії, статус патчу.
* **Executive Report:** Рівень ризику (Low/High), потенційний вплив на бізнес-процеси, фінансові ризики, резюме "одним абзацом".
* **Telegram Bot:** Канал для миттєвих алертів про критичні загрози.

## 5. Специфікація агентів (The Multi-Agent Core)

Кожен агент — це вузол у **LangGraph**.

1. **Ingestion Agent (Скаутинг):**
* *Дія:* Опитує джерела, перетворює HTML/Markdown у структурований JSON.
* *Tools:* `scrape_url`, `read_tg_history`, `rss_parser`.


2. **Triage Agent (Сортування):**
* *Дія:* Визначає "вагу" новини. Робить векторний пошук в OpenSearch, щоб перевірити, чи не писали про це раніше (дедуплікація).
* *Tools:* `opensearch_vector_search`.


3. **Infrastructure Correlation Agent (Архітектор):**
* *Дія:* Порівнює вразливість із Asset Inventory.
* *Logic:* Використовує Fuzzy Matching для версій ПЗ (наприклад, співставлення "v17.2" з "17.2.1a").
* *Tools:* `query_inventory_db`.


4. **Deep Research Agent (Координатор OpenClaw):**
* *Дія:* Асинхронно викликає OpenClaw для автономного вивчення загрози.
* *Tools:* `openclaw_execute_task`.


5. **Reporting Agent (Копірайтер):**
* *Дія:* Агрегує всі дані від попередніх агентів і формує фінальний текст.

## 6. Необхідні Tools (Специфікація інструментів)

Для Python розробника необхідно реалізувати:

* `InfrastructureSearchTool`: Пошук у OpenSearch індексі `assets`.
* `ThreatVectorStorage`: Запис нових загроз у `threat_intel_index`.
* `LiteLLMHandler`: Уніфікований клас для виклику моделей з підтримкою retry-логіки та підрахунку токенів.
* `TelegramNotifier`: Відправка форматованих повідомлень з кнопками "Підтвердити" або "Ігнорувати".

---

## 7. Нефункціональні вимоги

* **Privacy:** Використання локальних моделей (через LiteLLM -> Ollama) для обробки Asset Inventory. Конфіденційні дані не повинні залишати контур компанії.
* **Scalability:** Система повинна обробляти до 1000 повідомлень/джерел на годину.
* **Reliability:** Langfuse повинен зберігати повний trace кожного кроку для аудиту рішень ШІ.

## 8. Роадмап (Milestones)

1. **MVP 1 (Core):** Налаштування LiteLLM + FastAPI + Telegram Ingestion.
2. **MVP 2 (Intelligence):** LangGraph логіка + OpenSearch RAG для Asset Inventory.
3. **MVP 3 (Autonomous):** Інтеграція з OpenClaw та генерація звітів.
4. **Beta:** Розгортання адмінки та Guardrails.
